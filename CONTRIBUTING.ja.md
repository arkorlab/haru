# Haru への貢献

ご興味を持っていただきありがとうございます！ Haru は**初期段階**にあります。コアアイデア (Active/Standby な LLM 推論フリートのためのプロバイダー中立な GPU HAL) は、実際に運用するであろう人たちと一緒に設計したいものです。Issue、議論、PR のいずれも歓迎します。

[English version](CONTRIBUTING.md)

## 貢献の方法

| 労力         | 特に役立つこと                                                                       |
| ------------ | ------------------------------------------------------------------------------------ |
| **5 分**     | [垂直スライスのウォークスルー](README.ja.md#垂直スライスを試す-gpu-不要) を試して、混乱した点・壊れた点について [Issue を立てる](https://github.com/arkorlab/haru/issues/new)。 |
| **半日**     | 小さな PR (ドキュメント修正、エラーメッセージの改善、テスト付きの状態機械エッジケース追加) を送る。 |
| **継続的に** | 必要なプロバイダー、GPU レイアウト、フェイルオーバーポリシーを教えてください。優先順位付けに使います。 |

非自明な変更 (新しいドライバー、新しい昇格ステップ、ポリシー項目) のアイデアがある場合は、コードを書く前に状態機械と API の形について合意できるよう、まず Issue を開いてください。

具体的に着手できるタスクを探している場合は、[KNOWN_ISSUES.ja.md](KNOWN_ISSUES.ja.md) を見てください。レビュー済みで先送りした作業を、ファイル参照と修正方針付きで追跡しています。

## リポジトリ構成

```
haru/
├── packages/
│   ├── protocol/           # Zod スキーマ / 型付き API 契約 (型の単一情報源)
│   ├── core/               # 純粋な状態機械、昇格プランニング、route intent
│   ├── db/                 # Neon/Postgres 状態ストア (Drizzle、CAS リポジトリ、PGlite テスト)
│   ├── driver-skypilot/    # SkyPilot ドライバー境界
│   └── driver-skyserve/    # SkyServe ドライバー境界
├── services/
│   ├── haru-server/        # コントロール API + reconciler + OpenAI 互換 chat proxy
│   └── haru-supervisor/    # GPU ドメイン側スーパーバイザー (vLLM sleep/wake、学習)
└── turbo.json              # ビルド / テストのオーケストレーション
```

## 開発セットアップ

**Node.js 24 (できれば最新版)** と **pnpm 11.0+** を使用してください。

```bash
git clone https://github.com/arkorlab/haru.git
cd haru
pnpm install
pnpm build         # turbo run build (全パッケージをカバー)
pnpm test          # モノレポ全体の vitest (PGlite ベースの DB/server テストを含む)
pnpm typecheck     # モノレポ全体の tsc (TypeScript 7)
pnpm lint          # oxlint --type-aware --deny-warnings の後、型情報ベースの strict ESLint 10
pnpm format        # oxfmt --write (設定は oxfmt.config.ts)
pnpm format:check  # oxfmt --check。未整形ファイルがあると CI が失敗する
```

特定のパッケージで作業するには:

```bash
pnpm --filter @haru/server dev       # コントロールサーバーを tsx watch
pnpm --filter @haru/supervisor dev   # スーパーバイザーを tsx watch
pnpm --filter @haru/core test:watch  # 純粋な core を vitest watch
```

## テスト

すべて GPU・クラウドアカウント・稼働中のデータベースなしでテストできます:

- `@haru/db` と `services/haru-server` は、コミット済みの Drizzle
  マイグレーションを使ってインメモリ **PGlite** 上でスイートを実行します。
  全状態遷移を守る compare-and-swap SQL が実際に実行されます
  (並行実行時の勝者決定レースを含む)。
- ドライバーは注入可能な `exec` 境界に対してテストされます
  (argv の記録、タイムアウトの伝播、エラーマッピング)。`sky`
  バイナリは不要です。
- スーパーバイザーは fake fetch (vLLM 管理エンドポイント)、fake
  子プロセスハンドル、fake timers (SIGTERM → 猶予 → SIGKILL
  エスカレーション) でテストされます。

新しい I/O は同じ注入可能な境界の背後に置いてください。変更した
コードの隣に vitest ケースがあると助かります。状態機械の変更では、
網羅的な遷移テーブルテストを拡張してください。

## スタイル規約

- **整形は oxfmt が所有します** (空白、折り返し、クォート、末尾カンマ)。
  `pnpm format` を実行してください。
- **2 つのリンター、それぞれルート設定 1 つ**: `oxlint --type-aware`
  (高速、tsgo ベース) の後に型情報ベースの strict ESLint 10。
  どちらもリポジトリルートの設定を読みます。パッケージごとの設定では
  なくルートにオーバーライドを足し、インラインの `eslint-disable`
  よりも「なぜ」のコメント付きのスコープ付きオーバーライドを
  優先してください。
- ファイル名は kebab-case (lint で強制)。コードのコメントは英語で
  書きます。
- コードと文章でエムダッシュ (U+2014) は使いません。代わりにコロン、
  カンマ、丸括弧、スペース付きハイフンを使ってください (姉妹
  リポジトリ Arkor と共有している規約です)。

## プルリクエストのガイドライン

PR は粗くてもまず受け入れる方針です。小さな貢献も心から歓迎しており、「小さすぎて送るほどじゃない」ということは決してありません。**以下のいずれも PR を出さない理由にはしないでください:**

- **大きさは気にしないでください。** 巨大な差分でも構いません。必要ならこちらで分割します。
- **説明が雑でも OK。** 説明が乱雑だったり薄かったりしても、PR が無いよりはずっと良いです。不明点はレビューで質問するので、それで PR を差し戻すことはしません。
- **テストは必須ではありません。** あると助かりますが (上記参照)、ブロッカーではありません。マージの一環としてこちらでテストを足すこともあります。
- **現段階では破壊的変更も問題ありません。** PR の説明にメモするだけで十分です。

## バグとセキュリティ問題の報告

- **バグ**: [GitHub Issues](https://github.com/arkorlab/haru/issues/new) で報告してください。再現手順、期待される動作と実際の動作、Node と pnpm のバージョンが添えられているとかなり助かりますが、「これが壊れている」という一行だけでも、報告しないよりはずっと良いです。
- **セキュリティ**: 公開 Issue を立てる代わりに、security@arkor.ai までメールしてください。48 時間以内に確認の返信をします。スーパーバイザーのトークン面、vLLM 管理エンドポイントの隔離、chat proxy のリクエスト処理に触れるものはすべて該当します。

## 行動規範

親切に、誠実だと仮定し、技術的な意見の相違は技術的なまま保ちましょう。それ以外 (ハラスメント、人格攻撃、排他的な振る舞い) は退場をお願いする理由になります。メンテナーの判断が最終です。
